## 1. main.rs：真正的程序启动入口

核心代码：

```rust
fn main() {
    if let Err(err) = aurora::run(std::env::args().nth(1)) {
        eprintln!("错误：{err}");
        std::process::exit(1);
    }
}
```

我的理解：

- `main()` 是 Rust 程序入口，`cargo run` 时最先执行这里
- `std::env::args().nth(1)` 取的是第一个用户传入参数
- 比如 `cargo run ./my-workspace` 时：
  - 第 0 个参数是程序本身
  - 第 1 个参数是 `./my-workspace`
- 所以这里拿到的是 `Option<String>`

这段代码的关键点有两个。

### 1.1 `if let Err(err) = ...`

这是 Rust 很常见的错误处理写法，意思是：

- 如果 `aurora::run(...)` 成功，就什么都不做
- 如果失败，就把错误拿出来处理

### 1.2 `eprintln!` 和 `std::process::exit(1)`

- `eprintln!` 是打印到标准错误输出
- `exit(1)` 表示异常退出
- `0` 一般表示正常退出
- `1` 一般表示出错退出

所以 `main.rs` 的职责其实很单纯：

- 调用统一入口 `aurora::run(...)`
- 如果失败，就打印错误并退出

## 2. lib.rs：项目总入口 / 模块中转站

我现在对 `lib.rs` 的理解是：它负责把模块组织起来，然后暴露一个统一的 `run()` 给 `main.rs` 调用。

核心思路大概是这样：

```rust
pub fn run(workspace_arg: Option<String>) -> Result<(), String> {
    let config = config::load_config(workspace_arg)?;
    cli::run(&config)?;
    Ok(())
}
```

这里最重要的是 `?`。

它的意思是：

- 如果结果是 `Ok(...)`，就继续往下执行
- 如果结果是 `Err(...)`，就立刻把这个错误从当前函数返回出去

所以：

```rust
let config = config::load_config(workspace_arg)?;
```

差不多等价于：

```rust
let config = match config::load_config(workspace_arg) {
    Ok(value) => value,
    Err(err) => return Err(err),
};
```

也就是说，`lib.rs` 现在做的事情很明确：

- 先加载配置
- 配置失败就直接退出
- 配置成功再进入 CLI

所以它像一个“总调度入口”，但还不是完整的业务逻辑层。

## 3. cli.rs：真正的命令行交互层

这几个文件里，`cli.rs` 是我一开始看得最多的一个。

我现在对它的理解是：它负责终端界面和输入输出，但不直接负责 AI 逻辑本身。

它主要做这些事：

- 显示启动 Banner
- 读取用户输入
- 调用 `App` 处理输入
- 打印回复
- 处理 `quit / exit / /clear`

### 3.1 render_banner()

这个函数负责打印启动页面。

我注意到这里有几个点：

- `Path::new(&config.workspace)` 是把路径转成 `Path`
- 这样后面就可以用 `workspace.display()` 比较好看地打印出来
- `render_banner(...)` 返回的是 `io::Result<()>`
- 但 `cli::run()` 返回的是 `Result<(), String>`

所以这里会用：

```rust
.map_err(|err| err.to_string())?
```

意思是把 `io::Error` 转成 `String`，方便统一错误类型。

### 3.2 `write!` 和 `flush()`

这里我也顺手理解了一下：

- `write!` 可以把内容写到指定输出目标，比如 `stdout`
- `println!` 更像是直接打印一整行
- `stdout.flush()?` 的意思是强制立刻把缓冲区内容输出到终端

这个在提示符场景里很重要，比如：

```rust
write!(stdout, "> ");
stdout.flush()?;
```

因为提示符没有换行，如果不 `flush()`，有时不会马上显示。

### 3.3 CLI 和 App 已经开始分层

这一段我觉得是最关键的理解之一：

```rust
let mut app = App::new(config.clone(), OllamaChatClient);
```

这说明现在的结构不是“CLI 直接调模型”，而是：

- `cli.rs` 只管终端输入输出
- `app.rs` 负责应用逻辑
- `ollama.rs` 负责模型调用

也就是说，CLI 已经开始从“主逻辑层”退出来了。

### 3.4 `trim()` 和输入清理

```rust
let trimmed = input.trim();
```

这个就是把前后空格和换行去掉。

比如：

- 用户输入：`hello\n`
- `trim()` 之后：`hello`

这是 CLI 里非常基础但必要的一步，因为终端输入天然会带换行。

### 3.5 `matches!` 和特殊命令判断

我还注意到这段：

```rust
let should_show_thinking =
    !trimmed.is_empty() && !matches!(trimmed, "quit" | "exit" | "/clear");
```

它的意思是：

- 如果输入不是空
- 并且也不是 `quit / exit / /clear`
- 才显示“助手> 正在思考...”

这个判断挺合理，因为：

- 空输入不需要思考
- 退出命令不需要思考
- 清空会话不需要思考

### 3.6 `match app.handle_text(...)`

这里能明显看出 CLI 只是“显示层”：

```rust
match app.handle_text(trimmed) {
    Ok(TurnOutcome::Ignored) => continue,
    Ok(TurnOutcome::Exit(message)) => {
        println!("{message}");
        break;
    }
    Ok(TurnOutcome::Cleared(message)) | Ok(TurnOutcome::Reply(message)) => {
        println!("{message}");
    }
    Err(err) => {
        println!("助手> 执行失败：{err}");
    }
}
```

我的理解：

- `Ignored`：忽略，比如空输入，直接下一轮
- `Exit`：打印消息，然后 `break` 退出循环
- `Cleared / Reply`：都只是打印一条消息
- `Err(err)`：打印失败信息，但不会退出程序

这一点很重要：

即使某一轮失败，CLI 也会继续活着，等下一次输入。

## 4. app.rs：应用逻辑层

### 4.1 app.rs 在整条链路里的位置

`app.rs` 是目前这几个文件里最核心的应用逻辑层。

前面看的 `cli.rs` 只是负责：

- 读取用户输入
- 打印助手回复

但真正判断用户输入是什么意思、要不要退出、要不要清空会话、要不要调用模型，都是在 `app.rs` 里完成的。

### 4.2 `trait ChatClient`：定义“聊天客户端”的标准

```rust
pub trait ChatClient {
    fn chat(
        &mut self,
        ollama_url: &str,
        model: &str,
        history: &[ChatMessage],
        user_text: &str,
    ) -> Result<String, String>;
}
```

`trait` 可以理解成：

- 接口
- 能力标准
- 一种约定

这里 `ChatClient` 的意思是：

只要某个东西实现了 `ChatClient`，它就必须有一个 `chat()` 函数。

为什么这里要写 `trait`？

因为这样 `App` 不会直接绑定死 Ollama。

现在你用的是：

- `OllamaChatClient`

但以后也可以换成：

- `OpenAIChatClient`
- `ClaudeChatClient`
- `MockChatClient`
- `TestChatClient`

只要它们都实现 `ChatClient`，`App` 就不用改。

这一点我觉得很像“面向接口编程”的感觉：

谁实现了这个 `trait`，谁就必须具备 `trait` 里规定的函数；这个“谁”可以是 `struct`，以后也可以是别的具体类型。

### 4.3 给 `OllamaChatClient` 实现 `ChatClient`

```rust
impl ChatClient for OllamaChatClient {
    fn chat(
        &mut self,
        ollama_url: &str,
        model: &str,
        history: &[ChatMessage],
        user_text: &str,
    ) -> Result<String, String> {
        crate::ollama::chat(ollama_url, model, history, user_text)
    }
}
```

意思是：

- `OllamaChatClient` 实现了 `ChatClient` 这个接口
- 所以它必须提供 `chat()` 函数

而它的 `chat()` 里面其实只是转发：

```rust
crate::ollama::chat(...)
```

所以真正和 Ollama 通信的代码应该在：

- `ollama.rs`

### 4.4 `TurnOutcome`：一轮输入之后的结果类型

`enum` 是枚举，表示一个值可能是几种情况之一。

这里 `TurnOutcome` 表示：

用户输入一轮之后，`App` 给 `CLI` 返回的结果类型。

有四种情况：

| 类型 | 什么时候出现 | 意思 |
| --- | --- | --- |
| `Ignored` | 用户输入空内容 | 忽略 |
| `Exit(String)` | 用户输入 `quit` 或 `exit` | 退出程序，并带一句退出提示 |
| `Cleared(String)` | 用户输入 `/clear` | 清空会话，并带一句提示 |
| `Reply(String)` | 普通聊天 | 返回助手回复 |

### 4.5 `#[derive(Debug, PartialEq, Eq)]` 是啥

```rust
#[derive(Debug, PartialEq, Eq)]
```

这是自动给 `TurnOutcome` 生成一些能力。

| derive | 作用 |
| --- | --- |
| `Debug` | 可以用 `{:?}` 打印调试信息 |
| `PartialEq` | 可以比较两个值是否相等 |
| `Eq` | 完整相等比较 |

### 4.6 `App<C>`：应用核心结构体

```rust
pub struct App<C> {
    config: AppConfig,
    session: Session,
    client: C,
}
```

这个就是应用核心。

它里面有三个东西：

| 字段 | 作用 |
| --- | --- |
| `config` | 保存配置 |
| `session` | 保存当前会话历史 |
| `client` | 负责调用模型的客户端 |

这里的 `C` 是泛型。

也就是说，`client` 不一定非得是 `OllamaChatClient`，可以是任何实现了 `ChatClient` 的东西。

### 4.7 `impl<C: ChatClient> App<C>`

这句有点绕，但很重要。

它的意思是：

只有当 `C` 实现了 `ChatClient`，这个 `App<C>` 才拥有下面这些方法。

为什么要限制？

因为下面的代码会调用 `self.client.chat(...)`。

如果 `C` 没有实现 `ChatClient`，那编译器就没法保证 `chat()` 一定存在。

### 4.8 `handle_text()` 的完整逻辑

我现在把它理解成：`app.rs` 里最核心的函数就是 `handle_text()`。

#### 第一步：清理输入

```rust
let trimmed = input.trim();
```

去掉前后空格和换行。

比如：

- `" hello \n"` 变成 `"hello"`

#### 第二步：判断是不是退出

```rust
if matches!(trimmed, "quit" | "exit") {
    return Ok(TurnOutcome::Exit("助手> 下次见。".to_string()));
}
```

如果用户输入：

- `quit`
- `exit`

就返回退出状态。

这里的：

```rust
matches!(trimmed, "quit" | "exit")
```

意思是：

`trimmed` 是否等于 `quit` 或 `exit`。

#### 第三步：普通聊天时调用模型

如果不是空输入、不是退出、也不是 `/clear`，那就会调用：

- `self.client.chat(...)`

也就是把这次输入交给模型客户端处理。

#### 第四步：保存这一轮对话

```rust
self.session.push_turn(trimmed, &reply);
```

这句是把用户输入和助手回复一起保存进 `session`。

也就是历史记录里会多一轮：

- `user: trimmed`
- `assistant: reply`

这样下一次调用模型时：

- `self.session.history()`

就会带上前面的聊天上下文。

### 4.9 一次正常聊天的流程

假设你输入：

`你好`

流程大概是：

```text
cli.rs 读取到 "你好"
↓
app.handle_text("你好")
↓
trim 后还是 "你好"
↓
不是空
↓
不是 quit / exit
↓
不是 /clear
↓
调用 self.client.chat(...)
↓
Ollama 返回 reply
↓
session.push_turn("你好", reply)
↓
返回 TurnOutcome::Reply("助手> xxx")
↓
cli.rs 打印出来
```

## 5. config.rs：启动时加载配置

这个文件的作用是：决定 Aurora 在哪个工作目录运行，用哪个模型，连接哪个 Ollama 地址。

### 5.1 `env::set_current_dir(&workspace)`

这一句很重要。

它会把当前进程的工作目录切换到 `workspace`。

也就是说，后面程序里用相对路径时，都是基于这个 `workspace`。

### 5.2 `load_dotenv(&workspace)?`

这句会加载 `workspace` 下面的 `.env` 文件。

所以 `config.rs` 的主线就是：

```text

读取 .env

```

它负责的是“启动配置入口”，不是聊天逻辑。

## 6. session.rs：保存聊天上下文

这个文件负责保存当前会话历史。模型之所以能理解“刚才那个”“继续说”，就是因为这里保存了历史消息。

### 6.1 `ChatMessage`

```rust
#[derive(Clone)]
pub struct ChatMessage {
    pub role: &'static str,
    pub content: String,
}
```

一条聊天消息有两个字段：

| 字段 | 作用 |
| --- | --- |
| `role` | 谁说的 |
| `content` | 说了什么 |

### 6.2 `push_turn()`

```rust
pub fn push_turn(&mut self, user_text: &str, assistant_text: &str) {
```

这个函数保存一轮对话。

它会先 push 用户消息：

```rust
self.history.push(ChatMessage {
    role: "user",
    content: user_text.to_string(),
});
```

再 push 助手消息：

```rust
self.history.push(ChatMessage {
    role: "assistant",
    content: assistant_text.to_string(),
});
```

然后：

```rust
self.trim_history();
```

限制历史长度。

所以一轮正常对话结束后，历史里会加两条消息。

### 6.3 `trim_history()`

```rust
fn trim_history(&mut self) {
    if self.history.len() > MAX_HISTORY_MESSAGES {
        let excess = self.history.len() - MAX_HISTORY_MESSAGES;
        self.history.drain(0..excess);
    }
}
```

如果历史超过 16 条，就删除最老的一部分。

比如现在有 20 条：

- `excess = 20 - 16 = 4`

然后：

```rust
self.history.drain(0..excess);
```

删除第 0 到第 4 条，也就是最早的 4 条。

这样就只保留最近的上下文。

## 7. ollama.rs：真正调用本地模型

这个文件负责把当前对话打包成 HTTP 请求，发给本地 Ollama 的 `/api/chat` 接口，然后解析返回内容。

### 7.1 构造 payload

```rust
let payload = json!({
    "model": model,
    "stream": false,
    "messages": messages,
    "options": {
        "temperature": 0.2
    }
})
.to_string();
```

这就是发给 Ollama 的 JSON。

大概长这样：

```json
{
  "model": "gemma4:e4b",
  "stream": false,
  "messages": [
    {
      "role": "system",
      "content": "你是 AuroraPulse..."
    },
    {
      "role": "user",
      "content": "你好"
    }
  ],
  "options": {
    "temperature": 0.2
  }
}
```

几个字段意思：

| 字段 | 意思 |
| --- | --- |
| `model` | 用哪个模型 |
| `stream: false` | 不使用流式输出，等完整回复 |
| `messages` | 对话消息数组 |
| `temperature: 0.2` | 回复更稳定、少发散 |

### 7.2 先读 Ollama 返回的响应

```rust
let mut response: String = String::new();

io::Read::read_to_string(&mut stream, &mut response)
    .map_err(|err| format!("读取 Ollama 响应失败：{err}"))?;
```

这里的意思是：

- 创建一个空字符串 `response`
- 从 `stream` 这根网络连接管道里读取数据
- 把读到的内容塞进 `response`

`stream` 就是前面通过：

```rust
TcpStream::connect(...)
```

连出来的网络连接。

### 7.3 把 HTTP 响应拆成 `head` 和 `body`

```rust
let (head, raw_body) = response
    .split_once("\r\n\r\n")
    .ok_or_else(|| "Ollama 响应格式不正确".to_string())?;
```

HTTP 响应大概长这样：

```text
HTTP/1.1 200 OK
Content-Type: application/json
Content-Length: 123

{"message":{"role":"assistant","content":"你好"}}
```

中间这个空行：

```text
\r\n\r\n
```

就是 header 和 body 的分界线。

切完之后：

- `head` = HTTP 响应头
- `raw_body` = JSON 内容

### 7.4 取 HTTP 状态行

```rust
let status_line = head
    .lines()
    .next()
    .ok_or_else(|| "Ollama 响应缺少状态行".to_string())?;
```

`head` 里面第一行一般是：

```text
HTTP/1.1 200 OK
```

这行就叫状态行。

### 7.5 检查状态码

```rust
if !status_line.contains(" 200 ") {
    return Err(format!("Ollama 返回错误：{status_line}"));
}
```

只要不是 `200`，就当成错误。

顺手记：

- `400`：请求写错了
- `404`：找的地址不存在
- `500`：服务器自己炸了
- `502`：中间网关找后端失败

### 7.6 `ok_or_else(...)`

它的意思是：

- 如果是 `Some(value)`，就变成 `Ok(value)`
- 如果是 `None`，就变成 `Err(...)`

比如：

```rust
.ok_or_else(|| "Ollama 响应里没有 message.content")
```

意思就是：

如果响应里没找到 `message.content`，那就报错。

## 8. 我给自己补的一组概念

这组概念我觉得很有助于把 `ollama.rs` 看顺：

```text
TcpStream
= 网络管子，负责连到 127.0.0.1:11434

HTTP
= 管子里传的请求/响应格式，有 head 和 body

head
= 元信息，比如状态码、Content-Type、是否 chunked

body
= 真正的数据，比如 Ollama 返回的 JSON

JSON
= body 里的结构化数据，里面有 message.content

message.content
= 最终模型回复文本
```

## 10. 现在的结论

今天最大的收获不是“记住了某几个 Rust 语法”，而是开始看到项目分层了：

- `main.rs`：启动程序
- `lib.rs`：统一入口
- `cli.rs`：终端交互层
- `app.rs`：应用逻辑层
- `session.rs`：保存上下文
- `config.rs`：加载启动配置
- `ollama.rs`：模型调用层

这说明 AuroraPulse 已经不是一个“所有逻辑都堆在 CLI 里”的小脚本了，而是在慢慢长成一个有分层的 Rust runtime。

对我自己来说，这一轮最重要的不是全记住，而是先把这条链路顺下来：

`main -> lib -> cli -> app -> session / config / ollama`
