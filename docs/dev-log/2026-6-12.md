# 2026-06-12 开发日志

## 今天主要做了什么

Day 2 主要完成了 `planner.rs` 这个文件。

它的核心作用是：**把模型输出的一段 JSON，解析成程序能理解的“下一步决策”。**

也就是说，它负责回答这个问题：

> 模型现在决定要干什么？

目前可能有 4 种决策：

- 直接聊天
- 调用工具
- 检索资料
- 向用户追问

这些分支都被定义在 `PlannerDecision` 里：

```rust
pub enum PlannerDecision {
    Chat { reply: String },
    Tool { tool_name: String, arguments: Value },
    Retrieve { retrieve_query: String },
    Clarify { clarify_question: String },
}
```

| 决策 | 意思 |
| --- | --- |
| `Chat` | 直接回复用户 |
| `Tool` | 调用某个工具 |
| `Retrieve` | 先去检索资料 |
| `Clarify` | 信息不够，先问用户一个澄清问题 |

注意：现在只是先定义了 `Tool` 这个决策类型，真正的 tool 层还没有写。也就是说，planner 现在只会判断“这件事以后应该走工具”，不会真的执行工具。

---

## 1. 这个文件为什么叫 planner

因为它不是最终执行层，而是规划层。

普通聊天逻辑可能是：

```text
用户输入
  ↓
直接发给模型
  ↓
模型回复
```

但是带 planner 的 agent 逻辑会变成：

```text
用户输入
  ↓
让模型先输出一个 JSON 决策
  ↓
planner.rs 解析这个 JSON
  ↓
程序根据决策决定下一步
```

比如：

```text
用户：帮我查一下当前目录里的文件
```

模型 planner 输出：

```json
{
  "mode": "tool",
  "tool_name": "list_files",
  "arguments": {
    "path": "."
  }
}
```

Rust 解析成：

```rust
PlannerDecision::Tool { ... }
```

然后后面的 executor 再真正执行工具。

所以这个文件是为以后做 agent、tool calling、retrieval 准备的。

## 2. 读取 mode

核心代码：

```rust
let mode = value
    .get("mode")
    .and_then(Value::as_str)
    .ok_or_else(|| "planner decision 缺少 mode".to_string())?;
```

这段是在取：

```json
{
  "mode": "chat"
}
```

里面的 `"mode"`。

它要求 `mode` 必须存在，而且必须是字符串。

如果没有：

```json
{
  "reply": "hello"
}
```

就会报错：

```text
planner decision 缺少 mode
```

---

## 3. 根据 mode 分支处理

核心代码：

```rust
match mode {
    "chat" => { ... }
    "tool" => { ... }
    "retrieve" => { ... }
    "clarify" => { ... }
    other => Err(format!("未知 planner mode：{other}")),
}
```

这就是这个文件最核心的分流逻辑。

也可以简单理解成：

```text
mode = chat      -> PlannerDecision::Chat
mode = tool      -> PlannerDecision::Tool
mode = retrieve  -> PlannerDecision::Retrieve
mode = clarify   -> PlannerDecision::Clarify
```

## 4. mode = "tool"

核心代码：

```rust
"tool" => {
    let tool_name = required_text(&value, "tool_name", "tool decision 缺少 tool_name")?;
    let arguments = value
        .get("arguments")
        .cloned()
        .unwrap_or_else(|| serde_json::json!({}));

    if !arguments.is_object() {
        return Err("tool decision 的 arguments 必须是 object".to_string());
    }

    Ok(Self::Tool {
        tool_name: tool_name.to_string(),
        arguments,
    })
}
```

这个分支表示：planner 决定要调用工具。

它需要类似这样的 JSON：

```json
{
  "mode": "tool",
  "tool_name": "read_file",
  "arguments": {
    "path": "src/main.rs"
  }
}
```

这里有个细节：

```rust
.unwrap_or_else(|| serde_json::json!({}))
```

意思是：如果没有 `arguments`，默认用空对象 `{}`。

所以这个也合法：

```json
{
  "mode": "tool",
  "tool_name": "list_files"
}
```

会变成：

```text
arguments = {}
```

## 5. 辅助函数 required_text

核心代码：

```rust
fn required_text<'a>(
    value: &'a Value,
    field: &str,
    missing_message: &str,
) -> Result<&'a str, String> {
    let text = value
        .get(field)
        .and_then(Value::as_str)
        .ok_or_else(|| missing_message.to_string())?;

    if text.trim().is_empty() {
        return Err(format!("planner decision 的 {field} 不能为空"));
    }

    Ok(text)
}
```

这个函数专门负责从 JSON 里取一个必填字符串字段。

比如：

```rust
required_text(&value, "reply", "chat decision 缺少 reply")
```

意思是：

- 从 `value` 里取 `reply`
- 如果 `reply` 不存在，报错
- 如果 `reply` 不是字符串，报错
- 如果 `reply` 是空字符串或者只有空格，也报错

## 6. 这个文件的数据流

完整数据流是：

```text
raw_json: &str
  ↓
serde_json::from_str(raw_json)
  ↓
Value
  ↓
读取 mode
  ↓
match mode
  ↓
根据不同 mode 检查必填字段
  ↓
返回 PlannerDecision
```

这里的 `raw_json` 不是用户直接输入的原话。

更准确地说：

```text
用户输入的原始文本
  ↓
发给 LLM，也就是 Ollama
  ↓
LLM 在 planner 模式下输出 JSON 格式的“决策”
  ↓
planner.rs 解析这段 JSON
```

也就是说，`planner.rs` 不是直接处理用户原话的。它处理的是 LLM 返回的一段“决策 JSON”。

用户说一句话之后，模型不再只是直接自然语言回答，而是可以先返回一个 JSON，告诉程序下一步该干嘛。`planner.rs` 就负责把这个 JSON 解析成 Rust 里的枚举。

## 7. 今天的理解

我现在对 planner 的理解是：

> planner 就是把模型的自由输出，收束成程序能验证、能分流的结构化决策。

它目前看起来像“不同模式的判断”，但更准确一点说，它包括两件事：

- 判断这次应该走哪种 mode
- 抽出这个 mode 需要的字段

今天这一步还没有让 AuroraPulse 真正执行工具，但它已经把后面做 tool layer、retrieval 和 clarification 的入口先准备好了。
