# AuroraPulse 路线图

## 1. 路线图目标

这份路线图的目标，不是把 AuroraPulse 一次性做成一个“万能助理”，而是按阶段把它做成一个真正能日常使用的本地助理。

当前最核心的主线是：

语音输入 -> 本地模型理解 -> 工具执行 -> 返回结果

第一阶段先围绕 Spotify 跑通这条链路，后面再逐步扩展到更多日常工具和本地知识能力。

## 2. 总体阶段划分

AuroraPulse 的开发建议分为六个阶段：

1. `Phase 0`：稳定现有 CLI + Spotify Demo
2. `Phase 1`：抽出通用 Harness 和 Tool Registry
3. `Phase 2`：接入语音输入，做 Push-to-Talk Demo
4. `Phase 3`：补齐本地日常工具，形成最小助理雏形
5. `Phase 4`：接入本地 RAG，让助理具备“懂我的资料”的能力
6. `Phase 5`：接入 MCP 和更完整的语音体验

这六个阶段是从“能跑”到“能用”，再到“有扩展性”的路线。

## 3. Phase 0：稳定现有音乐控制 Demo

### 目标

先把当前已有的 `CLI + Gemma + Spotify` 这一条链路彻底稳定下来。

### 要做的事情

- 明确支持的意图类型
- 统一模型输出格式
- 把 Spotify 控制行为整理清楚
- 补齐错误处理
- 保证常见指令在 CLI 下稳定可用

### 第一版支持的指令

- 播放某首歌
- 播放某位歌手
- 暂停
- 继续
- 下一首
- 上一首
- 调节音量

### 本阶段交付物

- 一个稳定的 CLI 版本
- 一套固定的意图 JSON 输出结构
- 一份清楚的错误提示逻辑

### 成功标准

- 常见音乐指令可以稳定识别
- Spotify 能稳定执行对应动作
- 出错时用户能知道问题出在哪里

## 4. Phase 1：抽出通用 Harness

### 目标

把现在偏“写死在音乐场景里”的逻辑，重构成一个通用的 Agent 运行骨架。

### 要做的事情

- 新建 `harness` 作为主调度器
- 把 `planner` 和 `tool execution` 解耦
- 引入 `tool registry`
- 让系统支持 `chat / tool / clarify` 这类统一决策模式
- 为后面的语音、RAG、更多工具打基础

### 建议新增模块

- `aurorapulse/core/harness.py`
- `aurorapulse/core/tools.py`
- `aurorapulse/core/schemas.py`
- `aurorapulse/core/session.py`

### 本阶段交付物

- 一个不依赖 Spotify 业务细节的主循环
- 一个可以注册多个工具的工具层
- 一套稳定的 planner 输出 schema

### 成功标准

- 增加一个新工具时，不需要重写主流程
- 模型负责理解，Harness 负责控制，工具负责执行，职责边界清晰

## 5. Phase 2：语音 Push-to-Talk Demo

### 目标

做出第一个真正像助理的使用场景：

用户按下快捷键，说一句话，AuroraPulse 能听懂并在后台控制 Spotify。

### 要做的事情

- 接入全局快捷键
- 接入录音模块
- 接入本地 STT
- 把语音结果送进 Harness
- 返回简短反馈

### 推荐能力组合

- 快捷键监听
- 音频录制
- `faster-whisper`
- 现有本地 Gemma planner
- Spotify tool

### 本阶段交付物

- 一个 push-to-talk 的语音音乐助理 Demo

### 成功标准

- 用户按键即可开始说话
- 常见音乐指令可被正确识别
- Spotify 可以在后台开始播放
- 从说完话到执行完成的延迟可以接受

## 6. Phase 3：最小日常助理雏形

### 目标

从“音乐控制器”升级为“日常助理的最小版本”。

### 优先增加的工具

1. `notes`
- 保存一条笔记
- 快速记录想法

2. `reminders`
- 创建提醒
- 查看今天提醒

3. `files`
- 搜索文件
- 打开文件或文件夹
- 简单总结指定文档

4. `apps`
- 打开常用 App

### 本阶段交付物

- 至少 3 类非音乐工具
- 一个更像“个人助理”而不是“音乐遥控器”的版本

### 成功标准

- 用户可以每天用它做几件真实小事
- 工具数量增加后，系统仍然稳定

## 7. Phase 4：本地 RAG

### 目标

让 AuroraPulse 不只是能执行，还能根据本地资料回答问题和做总结。

### 适合先接入的知识源

- 项目文档
- Markdown 笔记
- 本地文本资料
- 选定目录下的 PDF

### 要做的事情

- 建立文档摄取流程
- 做 chunk 和 metadata
- 建立向量索引
- 支持语义检索
- 把检索结果注入模型上下文

### 推荐实现方向

- embedding 模型本地化
- 向量库先选轻量方案
- 元数据可落 SQLite

### 本阶段交付物

- 一个可以回答“根据我的本地资料”的能力层

### 成功标准

- 文件与笔记类问题的回答明显变好
- 检索结果可追溯到原文来源

## 8. Phase 5：MCP 与完整语音体验

### 目标

在核心 Harness 稳定后，补齐更标准化的工具扩展能力和更自然的使用方式。

### MCP 的作用

- 接更多外部能力
- 让工具扩展更标准化
- 减少以后每个新集成都手写适配的负担

### 语音体验补齐方向

- TTS
- 更轻量的状态反馈
- 后台常驻
- 菜单栏入口
- 后期可选唤醒词

### 本阶段交付物

- MCP adapter
- voice-first 版本
- 更接近日常可持续使用的交互体验

### 成功标准

- 新工具接入成本明显下降
- 语音交互变成真正主入口

## 9. 每个阶段都应该坚持的工程原则

- 小模型少承担复杂推理
- Harness 多承担控制与兜底
- 工具边界要窄而清晰
- 高风险操作必须可控
- 优先做可用的单轮能力，不急着做复杂 autonomy
- 每加一层复杂度，都要先保证前一层是稳定的

## 10. 当前最推荐的执行顺序

如果按你现在的目标，我建议最近就按这条顺序推进：

1. 稳住当前 CLI 音乐控制链路
2. 抽出 Harness 和 Tool Registry
3. 做 Push-to-Talk 语音音乐 Demo
4. 增加 notes / reminders / files 三类工具
5. 再做本地 RAG
6. 最后接 MCP 和完整语音体验

## 11. 当前阶段结论

AuroraPulse 现在最合理的起点，不是直接做一个全能助理，而是先做一个：

本地模型驱动、语音可触发、能稳定调工具的执行型助理。

只要第一条主链路做顺了，后面不管是加 RAG、加更多工具，还是接 MCP，都会自然很多。
